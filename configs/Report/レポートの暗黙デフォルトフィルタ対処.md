# Salesforce Report の暗黙デフォルト timeFrameFilter 対処

**目的**: Salesforce Report は `<timeFrameFilter>` を XML で明示しないと、SF 側が勝手に `THIS_FISCAL_QUARTER`（今四半期）のデフォルト日付フィルタをかけてくる。これに気付かず「データがあるのに Metric が 0 件」状態でデモに出すと信頼を失う。

**更新**: 2026-05-04 初版。FlowOrchestrationDemo S-n5p「差し戻し 0 件なのに実際は 2 件」事故を受けた体系化。

---

## 1. 何が起きるか

`reports/*.report-meta.xml` に `<timeFrameFilter>` が **無い**と、そのレポートを runtime で実行すると裏側で以下が暗黙に効く:

```
StandardDateFilter[column=<baseObject 標準日付列>, durationValue=THIS_FISCAL_QUARTER, startDate=<四半期開始>, endDate=<四半期末>]
```

`column` は baseObject ごとに固定:

| baseObject | デフォルトで使われる日付列 |
|---|---|
| `Ringi__c`（カスタムの主要日付列がある場合） | `Drafted_Date__c` |
| `Ringi_Approver__c` | `Processed_At__c` |
| `Account` / `Opportunity` / `Case` 等標準 | `CreatedDate` |

App Builder / レポートビルダー画面で「日付範囲: 全期間」に見えても、**メタデータ XML でこれを明示していない限り暗黙デフォルトは効く**。

---

## 2. 検出方法（Apex Reports.ReportManager）

デプロイ済みレポートが runtime で何を効かせているかは、Apex から一撃で見える:

```apex
Reports.ReportResults res = Reports.ReportManager.runReport('00OxxxxxxxxxxxxMAK', true);
Reports.ReportMetadata md = res.getReportMetadata();
System.debug('DateFilter: ' + md.getStandardDateFilter());
System.debug('CustomFilters: ' + md.getReportFilters());
for (String k : res.getFactMap().keySet()) {
    if (k == 'T!T') System.debug('total: ' + res.getFactMap().get(k).getAggregates());
}
```

- `DateFilter.durationValue` が `THIS_FISCAL_QUARTER` / `THIS_MONTH` 等になっていて、そのレポートが意図していない場合は確実に罠
- `CUSTOM` で `startDate=null, endDate=null` なら「全期間」扱いで OK

レポート ID は以下で取れる:
```bash
sf data query --query "SELECT Id, DeveloperName FROM Report WHERE DeveloperName = 'My_Rejected_Ringi'" --target-org <alias>
```

## 3. 修正方法（XML 追記）

「現在状態スナップショット系」レポートには **`INTERVAL_CUSTOM`（開始・終了なし）** を明示する:

```xml
<timeFrameFilter>
    <dateColumn>Ringi__c$Drafted_Date__c</dateColumn>
    <interval>INTERVAL_CUSTOM</interval>
</timeFrameFilter>
```

### 重要な落とし穴

- `dateColumn` の区切り文字は **`$`** を使う（例: `Ringi__c$Drafted_Date__c`）
  - `.` 区切り（`Ringi__c.Drafted_Date__c`）で書くと `Invalid field name: Ringi__c.Drafted_Date__c` でデプロイ失敗
- `interval` の値は `INTERVAL_CUSTOM`（`CUSTOM` ではない。`INTERVAL_` プレフィックス必須）
- `Ringi_Approver__c` 系は `Processed_At__c`、`Ringi__c` 系は `Drafted_Date__c` を指定するのが定石

## 4. スナップショット／トレンドの判定ルール

| 用途 | 例 | timeFrameFilter |
|---|---|---|
| 現在状態スナップショット | 差し戻し中／承認待ち／下書き／回覧中／滞留 | `INTERVAL_CUSTOM` 明示 |
| 時系列トレンド | 月次起案 6 ヶ月／今月決裁額／今四半期予算消化 | `LAST_N_MONTHS:6` / `THIS_MONTH` / `THIS_FISCAL_QUARTER` 明示 |
| 抽象検索（リスト形式） | 「大型案件 Top10」「J-SOX 監査重要稟議」 | 金額・フラグ条件だけで十分。時間軸は `INTERVAL_CUSTOM` |

**共通ルール**: どれであっても、`timeFrameFilter` を **明示しないことだけは禁止**。暗黙デフォルトに委ねない。

## 5. ダッシュボード Metric が 0 件の時のチェックリスト

1. SOQL でデータが存在するか確認（`SELECT COUNT() FROM ... WHERE ...`）
2. レポート runtime の `DateFilter.durationValue` を Apex で確認
3. `THIS_FISCAL_QUARTER` / `THIS_MONTH` 等の暗黙フィルタが効いていたら、XML に `<timeFrameFilter>` 追記 → deploy
4. ダッシュボードキャッシュを強制 refresh（ダッシュ画面の「🔄」ボタン）
5. ブラウザキャッシュも疑うなら `Shift + Reload`

## 6. 過去の事故例

### 事故 1: セルフマネジメント Metric 4 本中 3 本が 0 件化

- **現象**: ユーザー「差し戻し 0 件なのに実際は 2 件あるじゃないですか」
- **真因**: PL-d24 が `Drafted_Date__c` を 2025-11〜2026-04 に 6 ヶ月分散 → デフォルト「今四半期 = 2026-04〜06」で過去月が落ちた
- **影響**: 差し戻し 2→0 / 承認待ち余裕あり 9→0 / 下書き 1→0 の 3 本が嘘表示。`期限 3 日以内` だけ偶然見えていたため気付かれにくかった
- **対応**: 4 本とも `INTERVAL_CUSTOM` 明示 → 実データ連動確認（差し戻し 2 / 期限 3 日以内 45 / 承認待ち 9 / 下書き 1）

### 事故 2: アストロ Round 5 #3「期限 3 日以内 16 と 承認待ち 16 が同値」

- **現象**: アストロが「中央 2 枚が同じ 16 で並ぶ → 定義が同じ？」と指摘
- **真因**: これも同じ暗黙デフォルトフィルタで両方が偶然同じ値になっていた
- **教訓**: 「定義を変える」方向で議論してしまい、真因はフィルタ問題だった。**違和感の真因を推測する前にメタデータ実態を Apex で見る**
